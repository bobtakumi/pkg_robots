## 1. 一言でいうと

**pkg_robots は、あなたの Obsidian ナレッジベース（Personal Knowledge Garden = PKG）を読み、
「自分の考え（zettel）」と「読んだ内容（文献ノート）」の間の見落とされたつながりを見つけて、
週に一度リンク候補を提案してくるロボット**です。判定には自宅の DGX Spark 上で動く日本語 LLM（LLM-jp-4）を使います。

ロボットは**提案するだけで、あなたのノートは書き換えません**（"read + propose"）。
採用するかどうか、実際にリンクを張るかどうかは、常にあなたが決めます。

## 2. なぜこれを作るのか（背景の物語）

これは Jorge Arango の記事 **"Robots in the Garden"** に出てくる「庭にロボットを迎える」構想の実装です。
Arango は PKG を**庭**にたとえます——一夜では実らず、手入れ（草むしり）を続けることで構造が育つもの。
その庭仕事を AI に手伝わせる9つの役割のうち、最初に実装するのが **Connector（つなぎ役）** です。

なぜ Connector が最初か。**このvaultを実測したら、深刻な「層の断絶」が見つかった**からです:

- 文献ノート（読んだ本の要約）は **675ファイル・約320万字**と充実している
- 自分の考えを書いた zettel は **124枚**ある
- ところが **zettel から文献ノートへのリンクはわずか3〜5本**。文献インデックスノート32枚の
  `zettel_linked` 欄は**全て空**。つまり「読んだこと」と「考えたこと」がほとんど繋がっていない
- さらに zettel の**約25%が孤立**（どこからもリンクされていない31枚、どこへもリンクしない34枚）

せっかく320万字の読書資産があるのに、自分の思考から参照する経路がない。
**この断絶を埋めるのが Connector の最初の仕事**であり、価値が出やすく効果も測りやすい。
だから「zettel × 文献ノート」の層をまたぐリンク提案に的を絞っています。

## 3. 頭に入れておく3つの原則

このプロジェクトの判断は、ほぼ全てこの3原則から導かれています。迷ったらここに戻ってください。

1. **read + propose（提案のみ）** — ロボットは vault を書き換えない。唯一の例外は提案レポート
   `_Reports/suggest-YYYYMMDD.md` を置くことだけ。人間の知的生産の領域（`2_Permanent/` の zettel）には触れない。
2. **precision > recall（量より質）** — このPKGは**過去に一度頓挫している**。「提案が多すぎて疲れる」のが
   頓挫の一因だった。だから週の提案は**上限5件・少数精鋭**（Arango の言う "Consistency over Intensity"）。
   良い候補を取りこぼしてでも、ノイズを出さないことを優先する。
3. **著作権の線引き** — 判定基準の元ネタである Arango の書籍 *Duly Noted* の原則文は、
   **所有者（あなた）が自分のコピーから書き写す**。Claude は本文を再生産しない。
   （`prompts/connector_judge.md` の Ch7 原則文は既にあなたが記入済み。）

## 4. どう動くのか（4段パイプライン）

ロボットは4つのコマンドを順に走らせるだけの、素朴な作りです。凝った常駐サービスではありません。

```
garden index      →  garden candidates  →  garden judge      →  garden report
（目：vaultを読む）   （当たりをつける）      （頭：LLMが判定）      （提案を書く）
```

1. **index** — vault の全ノートをベクトル化し、リンクグラフと統計を SQLite（`data/garden.db`）に作る。
   ロボットの「目」。日本語は素朴にやるとトークンが膨張して埋め込みモデルの上限を超えるので、
   専用の `bge-m3-8k` を使い、長いノートは分割する（詳細は §7 の落とし穴）。
2. **candidates** — LLM を使わず、純粋な類似度計算で「近い zettel × 文献」のペアを機械的に拾う。
   ここは安いので広めに拾い（top 30）、絞り込みは次段に任せる。
3. **judge** — 各ペアを DGX 上の LLM-jp-4 が読んで「繋ぐべきか/否か・関係の種類・根拠の逐語引用・確信度」を
   JSON で返す。ロボットの「頭」。根拠が原文の逐語引用でないと**捏造とみなして弾く**（幻覚ガード）。
4. **report** — 確信度でふるいにかけ、上位5件だけを提案レポートにする。vault に書き込む唯一の段。

## 5. 3台のマシン構成（なぜ分かれているか）

| マシン | 役割 | 補足 |
|---|---|---|
| **DGX Spark** | 判定 LLM（LLM-jp-4）をホスト | `spark-062c.local:11434`。llama.cpp。既存アプリ bobook も利用中 |
| **MBP（M3 Max・128GB）** | 埋め込みモデル＋**pkg_robots 本体** | ここで週次運用を回す。埋め込みは MBP ローカルの Ollama |
| **MacBook Neo** | 作業・チャット元 | これまでの開発・設計はここで行った。非力なのでモデルは動かさない |

- vault は Neo↔MBP を **MBP 上のベアリポジトリ（Tailscale）** で同期。
- pkg_robots（このリポジトリ）は **GitHub** で同期（`github.com/bobtakumi/pkg-robots`）。
- **なぜ MBP が本体か**: 埋め込みは MBP でホストと決めたため、埋め込みを使う index/candidates は
  MBP ローカルが速い。判定だけ DGX に投げる。→ だから今、Neo から MBP へ引っ越す（残タスク O11）。

## 6. これまでの意思決定の経緯（オープンクエスチョンの記録）

設計中に迷った点を「O番号」で記録し、実測や方針で1つずつ潰してきました。**解決済みの判断は覆さないこと**
（理由も含めて記録してあります）。詳細はプロジェクトノート（§9 参照）にありますが、要点だけ:

**解決済み:**
- **O1（埋め込みと判定の分担）** → 埋め込み=MBP、判定=DGX、ロボット本体=MBP。
- **O2（埋め込みモデル選定）** → `bge-m3-8k` を採用。較正データに対する再現率で決着:
  bge-m3-8k は recall@30=**63.8%**、日本語特化の ruri-large は **25.5%**（512トークン制約で細切れになり劣後）。
  残る約36%の取りこぼしは「語彙が重ならない概念的なつながり」で、純ベクトル検索の原理的限界。
  → これが将来 Phase 2 の GraphRAG で上乗せを狙う余地（63.8%が比較の基準値）。
- **O3（vaultの規模）** → 約850ノート・320万字。毎回ゼロから索引を作り直して十分な規模。
- **O10（LLM-jp-4 の構成）** → DGX は **llama.cpp llama-server**（Ollama でも vLLM でもない）、
  モデルは `llm-jp-4-32b-a3b-thinking-Q4_K_M.gguf`。**thinking（推論）モデル**だが、
  llama.cpp が推論を `reasoning_content` に分離するので、`content` には素の JSON が返る（好都合）。
  ただし `response_format: json_object` での JSON 強制は **400 で使えない** → プロンプトで JSON を強制し、
  失敗時はリトライで吸収する方式にした。
- **O8** → 採用したリンクの着地点として、文献インデックスノートの `zettel_linked` 欄を使う（記入は人間）。
- **O9** → コードは `~/pkg_robots`、GitHub で同期。

**未解決（今後の判断）:**
- **O11** → pkg_robots を Neo から MBP へ移設する（＝今まさにやろうとしていること）。
- **O4** → Recommender 役に Web 検索を足すか（Phase 1 のスコープ外）。
- **O6 / O7** → 別の役割（Validator の INT クラスタ、Adversary）の扱い。Phase 2 以降の論点。

## 7. 触る前に知っておくべき「落とし穴」

過去にハマった罠です。**これらを知らずにいじると壊れます**。

- **LLM-jp-4 は「つなげたがり」**。M6 回帰テストで、無関係なペアの **11/15 をリンクと誤判定**しました
  （Claude が判定役をやったときは 1/15 だった）。対策として **report は確信度≥5 でふるいにかけて**います
  （`config.toml` の `[report] min_confidence = 5`）。ふるい後の上位5件は全て的確でした。
  **この閾値を安易に下げないこと。** 提案洪水は頓挫の元です。詳細は `M6-回帰結果-LLMjp4.md`。
- **JSON を強制する設定は使えない**（DGX の llama.cpp が 400 を返す）。プロンプト側で JSON を促し、
  パース失敗はリトライで吸収する実装（`judge.py` の `judge_pair`）。ここは触らない。
- **逐語検証は幻覚ガード**。LLM が引用を言い換えると弾かれる（invalid になる）が、これは
  「根拠の捏造」を防ぐ正しい挙動。妥当率を上げたいからと検証を緩めないこと。
- **判定は遅い**（thinking モデルなので1件あたり数秒〜十数秒、稀にタイムアウト）。
  全候補を judge に流さず `--limit` で絞る。出力トークン上限は**指定しない**（絞ると推論で打ち切られ空になる）。
- **日本語のトークン膨張**。素朴に埋め込むと 3450 字程度で 8192 トークン上限を超える。
  だから `bge-m3-8k`（num_ctx 拡張）を使い、チャンクは2000字、超過分はクライアント側で段階的に切り詰める。

## 8. あなたが今やること / この先の道のり

**いま（O11: MBPへの移設と初回運用）:**
1. `HANDOFF-MBP.md` §3 を上から実行 — venv 作成 → Ollama + bge-m3-8k → `garden index` → `garden judge --regress` で回帰確認。
2. 回帰の数値が Neo での基準（JSON valid 29/35・gold 17/20・非gold link 11/15）と大きくズレないことを確認。
3. パイプライン一気通し（candidates → judge → report）で `_Reports/suggest-YYYYMMDD.md` を生成し、
   vault を commit/push して週末レビューで読む。

**この先（Phase 1 完了 → Phase 2）:**
- **M5 週次運用の定着**: 上のサイクルを週次で回す。採否を `data/decisions.jsonl` に記録すると、
  次回から却下ペアが自動除外され、採用率が閾値・プロンプト調整の根拠になる。
- **`garden stats`**（採用率の集計）と提案への安定ID付与 — まだ未実装。M5 で足す。
- **judge の精度追い込み（任意）**: プロンプトで「無関係なら迷わず skip」をさらに強調すれば
  素の precision も上がる。ただし確信度ふるいで実用水準には達しているので、運用を回してからで良い。
- **Phase 2**: GraphRAG / Reflector（草むしり）/ Mapper。index は既に孤立ノートやタグ頻度などの
  「庭の構造シグナル」を `data/stats.json` に出しているので、そこから着手できる。

## 9. 用語集と正典ドキュメント

**用語:**
- **zettel** — `2_Permanent/Zettelkasten/` にある、自分の考えを1枚1アイデアで書いたノート。
- **文献ノート** — `1_Literature/` にある、読んだ本・記事の要約。「00 タイトル」がインデックス、章ごとに分割。
- **層の断絶** — zettel 層と文献層がリンクで繋がっていない状態。Connector が埋める対象。
- **gold / 較正セット** — 判定の正解データ。人力で作った6/26レポートから抽出（`eval/gold_pairs.yaml`）。
  35ペアの較正セット（`eval/calibration_*`）は**凍結**してあり、モデルを差し替えたときの回帰テストに使う。
- **read + propose** — ロボットは読んで提案するだけ、書き換えない、の原則。
- **O番号** — 設計中のオープンクエスチョン（未決事項）の通し番号。§6 参照。
- **M番号** — 実装マイルストーン（M0環境〜M6配線）。
